# 文献搜索逻辑与开放获取策略

> 目的：为文献阅读 Agent 制定第一批 20–50 篇论文元数据的搜索策略，并明确开放 API、闭源全文、学校认证和合法下载边界。

---

## 1. 核心结论

文献搜索和全文获取需要分开设计：

1. **元数据搜索**：可以主要依赖开放 API，例如 OpenAlex、Semantic Scholar、Crossref、arXiv、NASA ADS。
2. **开放全文查找**：可以使用 Unpaywall、arXiv、出版社开放 API、机构仓储、EarthArXiv 等。
3. **闭源全文获取**：通常不能靠“设置一个 key”绕过权限；需要学校/机构订阅、出版社 API 权限、TDM 协议或用户手动下载。
4. **推荐第一版策略**：Agent 自动收集元数据和开放 PDF；遇到闭源论文，只保存 DOI、出版社链接和待下载状态，由用户通过学校认证或 Zotero 手动补充 PDF。

因此，第一版文献阅读 Agent 不应试图绕过权限，而应采用：

> 开放元数据优先 + 合法 OA PDF 自动获取 + 闭源 PDF 人工补充 + Zotero/本地文件夹接入。

---

## 2. 为什么不能简单用一个 key 访问所有闭源论文？

很多论文的元数据是开放的，但全文 PDF 受版权和订阅协议限制。API key 通常只能做以下事情之一：

- 提高开放元数据 API 的访问额度；
- 访问某个平台允许开放的全文；
- 在机构订阅范围内访问出版社全文；
- 使用出版社的 Text and Data Mining API，但仍受机构订阅、API 协议、速率和用途限制。

也就是说：

> API key 不是绕过 paywall 的万能钥匙。对于闭源论文，合法访问仍依赖机构订阅、用户登录、出版社授权或 TDM 协议。

对于本项目，合理做法是：

1. 自动查找开放版本；
2. 找不到开放版本时，记录为 `fulltext_status: closed_or_manual_required`；
3. 用户通过学校 VPN/图书馆/出版社页面手动下载；
4. 将 PDF 放入 `papers/raw_pdf/`；
5. Agent 再从本地 PDF 继续解析和阅读。

## 3. 推荐数据源与分工

### 3.1 OpenAlex

链接：[OpenAlex Developers](https://developers.openalex.org/)

用途：

- 大规模开放学术元数据；
- 根据关键词、作者、概念、年份、期刊等检索论文；
- 获取 DOI、标题、作者、摘要倒排索引、年份、期刊、引用数、开放获取状态等；
- 适合作为第一层广覆盖检索。

适合本项目的用法：

- 搜索主题相关候选论文；
- 获取开放获取状态；
- 按引用数、年份、主题过滤；
- 作为 Semantic Scholar / Crossref 的补充。

注意：

- OpenAlex 主要是元数据源，不保证直接提供 PDF；
- 摘要有时是 inverted index，需要恢复；
- 对具体全文仍需结合 Unpaywall 或出版社链接。

---

### 3.2 Semantic Scholar API

链接：[Semantic Scholar API](https://www.semanticscholar.org/product/api)

用途：

- 论文搜索；
- 标题、摘要、作者、年份、venue；
- 引用数、参考文献、被引文献；
- TLDR、fieldsOfStudy 等字段；
- 适合做相关性筛选和 citation graph 扩展。

适合本项目的用法：

- 第一批 20–50 篇候选论文搜索；
- 根据引用数、年份、摘要筛选；
- 从核心论文向前追参考文献、向后追引用文献；
- 配合 Screening Prompt 做初筛。

注意：

- 可能需要 API key 以获得更稳定额度；
- 主要是 metadata 和 discovery，不等于全文下载；
- 部分论文可能没有 DOI 或摘要。

---

### 3.3 Crossref REST API

链接：[Crossref REST API documentation](https://www.crossref.org/documentation/retrieve-metadata/rest-api/)

可参考文档：[Crossref REST API docs on GitHub](https://github.com/CrossRef/rest-api-doc)

用途：

- DOI 元数据权威查询；
- 根据题名、作者、期刊、年份检索 DOI；
- 获取出版信息、期刊、出版商、license、reference 等；
- 适合 DOI 校验和 metadata 补全。

适合本项目的用法：

- 根据标题补 DOI；
- 根据 DOI 校验论文元数据；
- 获取期刊、出版社、年份、license 信息；
- 和 OpenAlex/Semantic Scholar 结果去重。

访问方式和礼貌规范：

- 公共 REST API 通常不需要注册 token；
- 建议使用 HTTPS；
- 建议在请求中提供 `mailto` 或 User-Agent 中包含邮箱，以进入 polite pool；
- 大量请求要缓存、限速、监控响应头中的 rate limit。

---

### 3.4 Unpaywall

链接：[Unpaywall API](https://unpaywall.org/products/api)

字段说明参考：[Unpaywall API fields](https://support.unpaywall.org/support/solutions/articles/44001777288-what-do-the-fields-in-the-api-response-and-snapshot-records-mean-)

用途：

- 根据 DOI 查找合法开放获取版本；
- 判断论文是否 OA；
- 获取 OA 状态：`closed`、`green`、`gold`、`hybrid`、`bronze`；
- 查找 best OA location。

适合本项目的用法：

- 对候选论文 DOI 批量查询开放全文；
- 如果存在开放 PDF，则自动下载或保存链接；
- 如果没有开放版本，则标记为需要人工获取。

推荐字段：

```yaml
is_oa:
oa_status:
best_oa_location:
fulltext_status: open / closed_or_manual_required
```

注意：

- Unpaywall 提供的是合法开放获取路径，不是破解 paywall；
- 对没有 DOI 的论文不方便；
- 需要在请求中提供邮箱参数是常见用法。

---

### 3.5 arXiv

链接：[arXiv API](https://info.arxiv.org/help/api/index.html)

用途：

- 搜索 arXiv 预印本元数据；
- 获取标题、作者、摘要、分类、提交时间；
- 对 arXiv 论文可直接获取开放 PDF。

适合本项目的用法：

- 查找 deep learning、AI Agent、methodology 相关论文；
- 查找部分地震学/地球物理相关预印本；
- 获取完全开放的 PDF。

注意：

- 地震学和 GNSS 传统论文不一定在 arXiv 上；
- arXiv 更适合 AI/ML 方法类文献。

---

### 3.6 NASA ADS

链接：[NASA ADS API](https://ui.adsabs.harvard.edu/help/api/)

用途：

- 天文、物理、地球物理相关文献检索；
- 适合 geophysics、GNSS、seismology、space geodesy 等交叉方向；
- 可用于发现传统地球科学论文。

适合本项目的用法：

- 搜索 GNSS seismology、geodetic earthquake early warning、real-time GNSS 等关键词；
- 补充 OpenAlex/Semantic Scholar 对地球物理文献覆盖不足的问题。

注意：

- 通常需要 API token；
- 主要做 metadata/discovery，不等于全文下载。

---

### 3.7 出版社 API 与 TDM

#### Elsevier

链接：[Elsevier Developer Portal](https://dev.elsevier.com/)

要点：

- Scopus API 偏向 citation、bibliographic metadata、abstracts；
- ScienceDirect API 涉及 journal/book full text；
- 用户可申请 API key；
- 完整访问依赖所在机构是否订阅对应 Elsevier 产品；
- TDM 相关接口存在，但受使用条款和机构权限限制。

#### Springer Nature

链接：[Springer Nature API](https://dev.springernature.com/)

要点：

- Meta API：metadata 和 abstracts；
- Open Access API：开放获取出版物全文；
- Full Text API / TDM：订阅内容全文，通常需要 API key、订阅和条款约束。

结论：

> 出版社 API 可以作为后续增强功能，但第一版不应依赖它们。它们不能保证访问所有闭源全文，且需要机构订阅、API key 和条款合规。

---

### 3.8 Zotero

链接：[Zotero Web API](http://www.zotero.org/support/dev/web_api/v3/)

用途：

- 管理用户自己的文献库；
- 读取 Zotero 条目、附件、笔记、tag；
- 和本地/云端文献管理流程集成。

适合本项目的用法：

- 对于需要学校认证手动下载的 PDF，用户可以先放入 Zotero；
- Agent 后续从 Zotero 导出的 BibTeX/RIS/Better BibTeX 或本地 storage 中读取 PDF；
- Zotero 不适合作为主要“搜索全网论文”的工具，但适合作为人工补全文献的入口。

## 4. 第一批 20–50 篇论文的搜索策略

### 4.1 不建议只用一个长 query

不要直接搜索：

```text
HR-GNSS deep learning rapid large earthquake source characterization finite fault inversion earthquake early warning
```

原因：

- 太长的 query 会漏掉很多相关论文；
- 不同数据库分词和排序逻辑不同；
- 地球物理文献不一定使用 deep learning 术语；
- 传统 GNSS 论文可能用 geodetic earthquake early warning，而不是 HR-GNSS。

### 4.2 推荐拆成 5 条搜索线

#### 搜索线 A：HR-GNSS / real-time GNSS 地震预警

关键词：

```text
"high-rate GNSS" earthquake early warning
"real-time GNSS" earthquake early warning
"geodetic earthquake early warning"
"GNSS seismology" earthquake magnitude
```

目标：找到 GNSS 地震预警和快速震级估计的经典论文。

#### 搜索线 B：PGD scaling / Mw saturation

关键词：

```text
"peak ground displacement" earthquake magnitude GNSS
"PGD scaling" earthquake magnitude
"magnitude saturation" GNSS earthquake early warning
```

目标：找到传统 PGD scaling 和震级饱和问题相关论文。

#### 搜索线 C：G-FAST / BEFORES / 有限断层快速反演

关键词：

```text
G-FAST GNSS earthquake
BEFORES GNSS earthquake
"finite fault" "real-time GNSS"
"rapid finite fault inversion" GNSS earthquake
```

目标：找到快速有限断层反演和实时 GNSS 源参数估计方法。

#### 搜索线 D：深度学习地震震级/震源估计

关键词：

```text
"deep learning" earthquake magnitude estimation
"deep learning" earthquake source characterization
"neural network" earthquake magnitude estimation
"machine learning" earthquake early warning magnitude
```

目标：找到深度学习相关方法，即使不一定使用 GNSS。

#### 搜索线 E：多模态融合和物理约束

关键词：

```text
"GNSS" "strong motion" earthquake early warning
"multimodal" earthquake early warning
"physics-informed" earthquake source inversion
"physics-informed neural network" earthquake source
```

目标：寻找 GNSS + 强震仪、多模态融合和物理约束建模相关文献。

## 5. 推荐检索流水线

### 5.1 第一轮：广覆盖 metadata 搜索

对每条搜索线：

1. 用 OpenAlex 搜索 20–50 条；
2. 用 Semantic Scholar 搜索 20–50 条；
3. 用 Crossref 补 DOI 和期刊信息；
4. 对地球物理关键词用 NASA ADS 补充；
5. 合并去重。

合并后预计得到 100–200 条候选元数据。

### 5.2 第二轮：自动初筛

对候选论文运行 Screening Prompt，输出：

```yaml
relevance_score:
reading_priority:
decision:
matched_topics:
next_action:
```

保留：

- `relevance_score >= 4` 的论文；
- 经典方法论文；
- 高引用论文；
- 与 GNSS/HR-GNSS/地震预警/震源表征强相关论文；
- 与深度学习方法强相关论文。

目标：筛到 20–50 篇。

### 5.3 第三轮：开放全文检查

对有 DOI 的论文：

1. 查询 Unpaywall；
2. 检查 arXiv / EarthArXiv / author manuscript；
3. 若存在合法 OA PDF，保存 PDF URL；
4. 若没有，标记为人工下载。

建议字段：

```yaml
fulltext_status: open_pdf_found / open_landing_page_found / closed_or_manual_required / no_doi
pdf_url:
oa_status:
manual_download_note:
```

### 5.4 第四轮：人工补全文献

对于核心但闭源的论文：

1. 用户通过学校认证、图书馆、出版社网页或 Zotero 手动下载；
2. 放入 `papers/raw_pdf/`；
3. 在 metadata 中记录本地 PDF 路径；
4. Agent 继续解析和精读。

## 6. 对闭源论文的处理策略

建议把论文分成四类：

| 类型 | 处理方式 |
|---|---|
| Open PDF | Agent 自动下载或记录 PDF URL |
| Open landing page | 保存页面和 DOI，必要时人工确认 PDF |
| Closed but institutionally accessible | 用户通过学校认证手动下载，放入本地目录 |
| Closed and inaccessible | 只保存 metadata，不进入全文精读 |

元数据字段：

```yaml
access:
  is_oa:
  oa_status:
  best_oa_location:
  pdf_url:
  landing_page_url:
  fulltext_status:
  local_pdf:
  manual_required:
```

---

## 7. 初始搜索结果数量建议

建议不要一开始就只收 20 篇，因为筛选后可能剩不多。更合理的是：

1. 每条搜索线抓取 30–50 条；
2. 五条线合并后约 150–250 条；
3. 去重后约 100–180 条；
4. 初筛后保留 30–50 条；
5. 精读优先级最高的 3–5 条。

如果只想小规模测试：

1. 每条搜索线抓取 10 条；
2. 合并后约 30–50 条；
3. 初筛后保留 10–20 条；
4. 精读 3 篇。

## 8. 推荐第一版实现方案

第一版不需要接出版社闭源 API，建议先做开放元数据和开放 PDF 检索。

### 8.1 输入

```yaml
topic: HR-GNSS + deep learning for rapid large-earthquake source characterization
search_lines:
  - HR-GNSS / real-time GNSS earthquake early warning
  - PGD scaling / magnitude saturation
  - G-FAST / BEFORES / finite fault inversion
  - deep learning earthquake magnitude/source characterization
  - multimodal / physics-informed earthquake early warning
```

### 8.2 输出

```text
papers/candidates.jsonl
papers/screening_results.md
papers/selected_papers.yaml
```

## 9. 对当前项目的建议

我建议后续按如下顺序执行：

1. 先写一个 `search_papers.py`，只做 metadata 搜索，不下载 PDF；
2. 先接 OpenAlex 和 Crossref，因为它们开放、稳定、无需复杂认证；
3. 再接 Semantic Scholar，如果需要则申请 API key；
4. 对每篇有 DOI 的论文查 Unpaywall；
5. 输出 `papers/candidates.jsonl`；
6. 用 Screening Prompt 做初筛，生成 `papers/screening_results.md`；
7. 对 `keep` 且有开放 PDF 的论文再下载；
8. 对 `keep` 但闭源的核心论文，交给用户手动下载。

这个逻辑最稳妥，也最符合法律和学术规范。

---

## 10. 一句话总结

文献搜索 Agent 的正确路线不是“用 key 绕过闭源论文”，而是：

> 用开放 API 做广覆盖元数据搜索，用 Unpaywall/arXiv 查找合法开放全文，把闭源核心论文标记为人工下载，再把用户合法获取的本地 PDF 接入精读和 RAG 流程。
